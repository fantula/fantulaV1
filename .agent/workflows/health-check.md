# 🩺 Health Check 技能脚本

> **使用前提**: 已读 `docs/SKILLS_HUB.md`，确认走此路线。
> **执行方式**: 从 Step 0 开始，逐步执行，每步完成后打 ✅，遇 ⛔ 必须停止等待用户确认。
> **适用场景**: 定期系统巡检 / 上线前验收 / 用户说"全量体检" / 怀疑系统状态异常

---

## Step 0: 明确检查范围

告知用户将执行以下三个维度的检查，并确认是否全部执行或只选部分：

```
✅ 维度 A：系统基础设施（数据库/R2/网关/网络）
✅ 维度 B：后端代码质量（Edge Functions）
✅ 维度 C：前端代码质量（后台 + PC端 + 移动端）
```

> 默认全部执行。用户可说"只检查后端"等进行范围限定。

---

## Step 1: 维度 A — 系统基础设施检查

### A1. 管控台状态灯检查

- [ ] 引导用户访问后台 `/admin/backend-settings` 页面
- [ ] 确认以下状态灯全部为绿色：
  - Database 连接状态
  - R2 存储状态
  - Gateway 状态
  - Internet 连通状态

🔴 如有红灯 → 记录在报告中，提示用户排查基础设施（不是代码问题）

### A2. 一键体检脚本

- [ ] 在终端执行：
  ```bash
  sh scripts/test_system_health.sh
  ```
- [ ] 记录输出的 HTTP 状态码和延迟数据

### A3. 近期错误日志检查

- [ ] 查询 Supabase `error_logs` 表：
  ```sql
  SELECT * FROM error_logs ORDER BY created_at DESC LIMIT 10;
  ```
- [ ] 记录近期错误模式（如有重复出现的错误类型）

**A 维度小结**：输出状态报告（🟢 正常 / 🟡 警告 / 🔴 故障）

---

## Step 2: 维度 B — 后端代码质量检查

> 按照 `docs/AI_TASK_DISPATCH.md` §后端SCAN流程 执行

- [ ] 列出所有 Edge Function 目录：`ls supabase/functions/`
- [ ] 逐函数检查：
  - [ ] 入口文件是否有完整的 `try/catch`
  - [ ] 是否调用了错误记录（Logger.error 或等效方案）
  - [ ] CORS 响应头是否在多函数中重复定义（应提取到 `_shared/`）
  - [ ] AWS SigV4 amzDate regex 是否正确（`/[:-]|\\.\\d{3}/g`）
  - [ ] 环境变量是否使用 `Deno.env.get('KEY')`，无硬编码
  - [ ] 函数是否仍被前端代码调用（grep 验证）
  - [ ] `_shared/` 中的工具函数是否有无引用的死代码

**B 维度小结**：输出问题清单（🔴/🟡/🟢 分级）

---

## Step 3: 维度 C — 前端代码质量检查

> 按照 `docs/AI_TASK_DISPATCH.md` §SCAN流程 执行，分三个模块

### C1. 后台管理端检查

- [ ] 抽查高频使用模块（orders / products / CDK / coupons / users）
- [ ] 使用后台 SCAN 清单检查：
  - AdminDataTable / AdminDataDialog / useAdminDialog 使用情况
  - adminRoute() 路由规范
  - console.log 守卫情况
  - API 调用的 try/catch + ElMessage
  - 组件 import 但未在 template 渲染的情况

### C2. PC 客户端检查

- [ ] 抽查核心路径（首页 / 商品详情 / 结算 / 个人中心）
- [ ] 使用客户端 SCAN 清单检查：
  - 路由是否使用 `pcRoutes.xxx()`
  - 价格是否格式化 `Number(price).toFixed(2)`
  - 关键按钮的 loading/disabled/spinner 状态

### C3. 移动端检查

- [ ] 抽查核心路径（首页 / 商品详情 / 购物车 / 结算）
- [ ] 重点检查 §模式1（滚动容器）：
  ```
  grep -r "min-height: 100vh" nuxt-frontend/pages/mobile/
  ```
  → 只有 min-height 而无 overflow-y:auto 的文件均为 Bug

**C 维度小结**：输出问题清单（🔴/🟡/🟢 分级）

---

---

## Step 5: 按用户指示执行修复（可选）

用户确认报告后，可选择：

**选项 A**: 立即修复 🔴 级别问题
→ 对每个 🔴 问题，走对应的 `bug-fix.md` 流程

**选项 B**: 只修复指定问题
→ 用户指定问题编号，走对应流程

**选项 C**: 生成修复计划，稍后执行
→ 输出每个问题的修复建议，用户自行安排时间

**选项 D**: 不修复，仅归档报告
→ 将报告追加到 `docs/SCAN_AUDIT_LOG.md`

---

## Step 6: 归档检查记录

无论是否修复，都应将报告追加到 `docs/SCAN_AUDIT_LOG.md`：

```markdown
## 健康检查记录 — [日期]
- 检查范围: [全量/指定范围]
- 发现问题: [🔴 X个 / 🟡 X个 / 🟢 X个]
- 已修复: [是/否/部分]
- 结论: [系统状态评级]
```
