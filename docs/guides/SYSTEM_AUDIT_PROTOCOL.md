# 系统全景体检协议 (System Audit Protocol)

> **版本**: V1.0 | **更新时间**: 2026-02-14
> **适用场景**: 定期巡检、大版本发布前、接手新项目时
> **执行者**: AI 工程师 / 高级开发人员

---

## 🏥 协议概述

本协议定义了对 Fantula 系统进行"全身体检"的标准流程。体检不仅仅是查错，更是为了**确保系统架构的健康度、安全性以及多端的一致性**。

### 体检原则
1.  **全局视角**: 不只看单个页面，要看整体架构 (Store, Router, Middleware)。
2.  **端到端一致性**: 重点检查 Mobile 是否正确复用了 PC 的逻辑。
3.  **零副作用**: 体检过程**只读**，绝对禁止修改数据。
4.  **证据留存**: 必须输出体检报告。

---

## 📋 Phase 1: 核心基础设施体检 (Core Infra)

### 1.1 后端服务 (Backend/Supabase)
- [ ] **API 健康度**: 检查关键 API (`/api/client/config`, `/api/admin/auth/me`) 响应时间。
- [ ] **错误日志分析**: 检查 Supabase Logs，寻找 500 错误或异常 Pattern。
- [ ] **Advisors 检查**: 运行 `get_advisors` 获取 Security/Performance 建议。

### 1.2 数据库 (Database)
- [ ] **RLS 策略覆盖率**: 确保所有表都启用了 RLS (Row Level Security)。
- [ ] **策略逻辑审计**: 再次确认 `auth.uid()` 逻辑是否严密。
- [ ] **备份状态**: 确认最近一次自动备份的时间。
- [ ] **索引健康度**: 检查是否有全表扫描的慢查询。

---

## 🖥️ Phase 2: 前端工程架构体检

### 2.1 全局状态 (State Management)
- [ ] **Store 设计**: 检查 `stores/` 目录。
    - 是否有重复定义的 Store?
    - `user.ts` 是否正确处理了 Hydration (服务端到客户端的状态同步)?
    - Mobile 是否直接引用了 PC 的 Store (正确) 还是复制了一份 (错误)?
- [ ] **类型安全**: 运行 `npm run typecheck` (或 `vue-tsc`) 检查全局类型错误。

### 2.2 路由与鉴权 (Routing & Auth)
- [ ] **Middleware 覆盖**:
    - `/admin` 路由是否全部被 `mgmt-auth` 保护?
    - `/mobile/profile` 路由是否全部被 `client-auth` 保护?
- [ ] **跳转逻辑**: 检查 401/403 跳转是否流畅，无死循环。

---

## 💻 Phase 3: PC 客户端体检

### 3.1 核心业务流程
- [ ] **登录/注册**: 模态框唤起、验证码发送、Token 存储。
- [ ] **商品详情**: SKU 选择、价格计算、库存显示。
- [ ] **支付流程**: 订单创建、支付跳转、回调处理。

### 3.2 布局与响应式
- [ ] **骨架屏**: 数据加载时的 Skeleton 是否不跳动。
- [ ] **窗口缩放**: 缩放浏览器时布局是否崩坏。

---

## 📱 Phase 4: 移动端专属体检 (Mobile Check)

> **⚠️ 重点关注**: 移动端是"体检"的高危区，最容易出现逻辑复制的代码异味。

### 4.1 逻辑复用性审计 (Critical)
- [ ] **API 调用**: Mobile 页面是否**直接引入**了 `api/client/xxx`? (必须是)
- [ ] **Store 引用**: Mobile 页面是否**直接引入**了 `stores/xx`? (必须是)
- [ ] **Composable**: 是否使用了 `composables/client/useOrderDetail` 等共享逻辑?
- [ ] **❌ 禁止项**: 检查是否有 `const fetchGoods = async () => { ... }` 这种手写的重复请求逻辑。

### 4.2 UI 与交互体验 (Experience)
- [ ] **视口配置**: `viewport` meta 标签是否正确 (禁止缩放)。
- [ ] **安全区域**: 刘海屏/底部 Home 条是否做了 `safe-area-inset` 适配。
- [ ] **触摸反馈**: 按钮点击是否有 `:active` 态或波纹效果。
- [ ] **滚动体验**: 列表滚动是否流畅，是否有回弹 (iOS)。

### 4.3 过渡与动画 (Transitions)
- [ ] **页面切换**: 路由跳转是否有转场动画 (Slide/Fade)?
- [ ] **模态框**: 打开/关闭是否有刚性闪烁? (应有 Fade/Spring 动画)。
- [ ] **列表加载**: 是否有 `TransitionGroup` 或交错动画?

---

## 🛡️ Phase 5: 安全与权限专项 (Security)

- [ ] **API 暴露检查**: 确认没有将 Admin API 暴露给 Client。
- [ ] **Service Key 泄露**: 再次 grep 检查代码中是否有硬编码的 `service_role_key`。
- [ ] **XSS 防御**: 检查 `v-html` 的使用情况。

---

## 📄 体检报告模板 (Audit Report Template)

每次体检结束后，必须输出以下格式的报告：

```markdown
# 🏥 系统体检报告

**执行信息**
- **体检日期**: YYYY-MM-DD
- **执行人**: [AI Name]
- **体检范围**: [全站 / 仅移动端 / 仅后台]
- **代码版本**: [Git Hash / Branch]

## 1. 核心指标概览
| 指标 | 状态 | 备注 |
|------|------|------|
| 编译状态 | ✅ Passing | build time: 12s |
| 类型检查 | ⚠️ 3 Errors | 忽略非关键类型错误 |
| API 健康 | ✅ Healthy | 均值 < 200ms |

## 2. 发现的问题 (按严重程度)

### 🔴 严重/阻断 (High)
1. **[位置]**: [描述]
   - *影响*: [后果]
   - *建议*: [修复方案]

### 🟡 警告/优化 (Medium)
1. **[位置]**: [描述]
   - *建议*: [修复方案]

### 🔵 建议/体验 (Low)
1. **[位置]**: [描述] (例如: 移动端动画缺失)

## 3. 移动端复用性审计 (Mobile Specific)
- [x] 数据逻辑完全复用
- [ ] **异常**: 发现 `pages/mobile/goods.vue` 重写了请求逻辑 -> 需重构

## 4. 下一步行动计划
1. 修复 High 级别 Bug (立即)
2. 优化 Medium 级别体验 (本周内)
...
```

---

## 🔄 修复执行标准

一旦体检报告确认，执行修复时需遵循：
1. **按表执行**: 严格对应报告中的 Item。
2. **原子提交**: 修复一个问题，验证一个。
3. **回归测试**: 确保修复不破坏原有功能 (特别是 Mobile 修改不影响 PC)。
