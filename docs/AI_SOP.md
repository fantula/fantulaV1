# AI 助手标准操作流程 (SOP)

> **版本**: V1.0 | **更新时间**: 2026-02-04
> **适用对象**: 所有执行测试和优化的 AI 助手
> **约束级别**: 强制执行

---

## ⚠️ 重要警告

```
┌─────────────────────────────────────────────────────────────────┐
│                        严格约束声明                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 你必须严格按照本文档的流程执行，不可跳步                       │
│                                                                 │
│  2. 每个阶段必须输出规定格式的报告                               │
│                                                                 │
│  3. 未经用户确认，禁止执行任何代码修改                            │
│                                                                 │
│  4. 修改范围必须严格限制在用户批准的范围内                        │
│                                                                 │
│  5. 修改一个地方时，绝对不能影响其他地方                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 📋 工作流程概览

```
┌─────────────────────────────────────────────────────────────────┐
│                      完整工作流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Phase 1: 接收任务                                               │
│     ↓                                                           │
│  Phase 2: 阅读规范 (必须完整阅读)                                 │
│     ↓                                                           │
│  Phase 3: 扫描分析 (只读，不改任何代码)                           │
│     ↓                                                           │
│  Phase 4: 输出报告 (等待用户确认)                                 │
│     ↓ [用户确认后才能继续]                                       │
│  Phase 5: 制定修改计划 (精确到行)                                 │
│     ↓ [用户确认后才能继续]                                       │
│  Phase 6: 执行修改 (严格按计划，一次一处)                         │
│     ↓                                                           │
│  Phase 7: 验证测试                                               │
│     ↓                                                           │
│  Phase 8: 输出完成报告                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 🚨 Phase 1: 接收任务

### 输入格式

用户会告诉你要测试的目标，例如：
- "测试后台管理的订单管理页面"
- "测试 PC 端商品详情页"
- "检查移动端购物车页面"

### 你必须确认

```markdown
## 任务确认

我收到的任务是：
- 测试范围: [后台管理/PC客户端/移动端]
- 具体页面: [页面名称]
- 文件路径: [预估路径]

请确认是否正确？
```

**⚠️ 必须等待用户确认后才能继续！**

## 🔍 Phase 3: 扫描分析

### ⚠️ 重要约束

```
┌─────────────────────────────────────────────────────────────────┐
│  在这个阶段，你只能：                                            │
│  - 读取文件内容                                                  │
│  - 分析代码结构                                                  │
│  - 记录发现的问题                                                │
│                                                                 │
│  你绝对不能：                                                    │
│  - 修改任何代码                                                  │
│  - 创建任何文件                                                  │
│  - 删除任何内容                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 扫描检查清单

根据页面类型使用对应清单：

#### 后台管理页面检查清单
```markdown
### 组件使用检查
- [ ] 使用 AdminDataTable? (是/否/应使用)
- [ ] 使用 AdminDataDialog? (是/否/应使用)
- [ ] 使用 AdminActionCard? (是/否/应使用)
- [ ] 使用 useAdminDialog? (是/否/应使用)
- [ ] 使用 useAdminList? (是/否/应使用)
- [ ] 使用 useBizFormat? (是/否/应使用)
- [ ] 使用 useBizConfig? (是/否/应使用)

### 代码质量检查
- [ ] 有无 console.log? (有/无)
- [ ] 有无硬编码状态? (有/无)
- [ ] 有无重复代码? (有/无)
- [ ] TypeScript 类型完整? (是/否)

### 功能检查
- [ ] 列表加载正常?
- [ ] 分页正常?
- [ ] 新增功能正常?
- [ ] 编辑功能正常?
- [ ] 删除功能正常?
```

#### PC/移动端页面检查清单
```markdown
### 共享逻辑检查 (最重要!)
- [ ] 使用共享 API? (api/client/)
- [ ] 使用共享 Store?
- [ ] 使用共享 Composable?
- [ ] 无重复业务逻辑?

### 代码质量检查
- [ ] 有无 console.log?
- [ ] 有无硬编码?
- [ ] 类型完整?

### 移动端特别检查
- [ ] 无 Element Plus 组件?
- [ ] 与 PC 端逻辑一致?
```

## 📋 Phase 5: 制定修改计划

### ⚠️ 重要约束

```
┌─────────────────────────────────────────────────────────────────┐
│  修改计划必须：                                                  │
│                                                                 │
│  1. 精确到具体文件和行号                                         │
│  2. 明确说明改动前后的代码                                       │
│  3. 保证修改范围最小化                                           │
│  4. 不影响其他功能                                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```markdown

- [列出不会动的文件/功能]

## 修改项目

### 修改 1: [标题]

**文件**: `[路径]`
**行号**: [行号]

**当前代码**:
```vue
[当前代码]
```

**修改后代码**:
```vue
[修改后代码]

**修改原因**: [说明]

**影响范围**: [仅此处]

---

### 修改 2: [标题]
...

## 修改顺序

1. 先修改 [xxx]
2. 再修改 [xxx]
3. ...

## 🔧 Phase 6: 执行修改

### ⚠️ 严格约束

```
┌─────────────────────────────────────────────────────────────────┐
│                       执行修改规则                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 一次只改一个地方，改完验证后再改下一个                        │
│                                                                 │
│  2. 严格按照确认的计划执行，不可自行增加修改                      │
│                                                                 │
│  3. 如果发现需要额外修改，必须停止并询问用户                      │
│                                                                 │
│  4. 修改后立即验证该处功能是否正常                               │
│                                                                 │
│  5. 如果修改导致问题，立即回滚并报告                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 执行日志格式

```markdown

## 📚 附录: 各端专属约束

### 后台管理专属约束

1. 必须使用全局组件 (AdminDataTable, AdminDataDialog 等)
2. 必须使用 Composable (useAdminDialog, useAdminList)
3. 必须使用 useBizFormat/useBizConfig
4. 禁止硬编码状态文案

### PC 客户端专属约束

1. 逻辑必须能被移动端复用
2. 订单详情页需特别注意（最复杂）
3. 禁止创建移动端无法复用的逻辑

### 移动端专属约束

1. **只改 UI 布局，不改业务逻辑**
2. 必须复用 PC 端的 API/Store/Composable
3. 禁止使用 Element Plus
4. 禁止新增 API 调用
5. 禁止复制 PC 端代码
